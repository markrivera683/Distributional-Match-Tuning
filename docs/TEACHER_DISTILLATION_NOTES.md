# Teacher Distillation Notes

This note records implementation-facing decisions for teacher distillation in
the current EBFT distribution-matching branch.

## 1. Two Routes

### Route A: API teacher

- teacher completions come from remote HTTP service;
- local pipeline handles tokenization, critic embedding, and reward.

### Route B: open-weight teacher

- teacher model is loaded as local actor group;
- teacher completions are generated inside the same training environment.

Both routes can feed the same target-measure interface in principle. Their
engineering properties are different.

## 2. Comparison

### Cost

- API teacher:
  - potentially high request cost and latency;
  - cost scales with sampled completions per block/prompt.
- open-weight teacher:
  - high local GPU memory/compute cost;
  - lower marginal cost once cluster is provisioned.

### Controllability

- API teacher:
  - limited access to logits/hidden states in most setups;
  - weaker reproducibility under backend changes and rate limits.
- open-weight teacher:
  - full control over generation config, seeds, and model version;
  - easier controlled ablations and failure diagnosis.

### Experimental value

- API teacher:
  - valuable for practical black-box distillation scenarios;
  - useful to test portability to proprietary teachers.
- open-weight teacher:
  - higher value for first-stage method validation;
  - cleaner causal attribution for algorithmic gains.

## 3. Current Repository Fit

### API teacher support

- status: **supported** ✔
- evidence:
  - `teacher_backend=remote`;
  - `teacher_provider` integration;
  - API style support (`completions` and `chat_completions`);
  - retry/cache controls in provider path.

### Multi-sample teacher support

- status: **supported** ✔
- evidence:
  - `cf_teacher_n_samples`;
  - per-prompt/per-block teacher sampling path;
  - mixed target construction in CF target builder.

### Teacher logprob / hidden-state support

- status: **not supported in current teacher API path** ✗
- current behavior:
  - API path consumes teacher completion text only;
  - hidden states/logprobs are generated by local critic after tokenization,
    not provided by remote teacher model.

### Target integration depth

- status: **partial** ◐
- current scope:
  - active in `cf_l1oo` teacher target mode;
  - not generalized across all reward variants.

## 4. Recommended Priority

Recommended order for the next implementation stage:

1. prioritize **open-weight teacher** as the primary distillation line;
2. keep API teacher as an extension path and robustness check;
3. finalize fair ablation matrix before broader feature-geometry changes.

Why this order:

- better reproducibility;
- better observability;
- lower risk of conflating algorithm changes with remote-service noise.

## 5. Relation to EMA Teacher

`EMA teacher` and `external teacher distillation` are not the same object.

### EMA teacher (self-distillation flavor)

- teacher is an exponential-moving-average copy of student/critic components;
- primary role is stabilization and temporal smoothing;
- no external capability source.

### External teacher (true distillation)

- teacher is a stronger external policy/model;
- provides additional target distribution support beyond dataset-only GT;
- primary role is capability transfer and richer conditional supervision.

Current repo uses EMA mainly for stabilization. Teacher-target mode should be
treated as a separate distillation axis, not as a synonym of EMA usage.
